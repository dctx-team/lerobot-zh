# 机器人与遥操作的 Processor

本指南展示如何构建和修改连接遥操作器（例如手机）到机器人和数据集的处理管道。Pipeline 标准化不同 action/observation 空间之间的转换，因此您可以交换遥操作器和机器人而无需重写粘合代码。

我们使用手机到 SO-100 follower 的示例来说明，但相同的模式适用于其他机器人。

**您将学到什么**

- 绝对 vs. 相对 EE 控制：每种的含义、权衡以及如何为您的任务选择。
- 三管道模式：如何映射 遥操作 action → 数据集 action → 机器人命令，以及 机器人 observation → 数据集 observation。
- 适配器（`to_transition` / `to_output`）：这些如何将原始字典转换为 `EnvTransition` 并返回以减少样板代码。
- 数据集特征契约：步骤如何通过 `transform_features(...)` 声明特征，以及如何聚合/合并它们以进行记录。
- 选择表示：何时存储关节、绝对 EE 姿态或相对 EE 增量——以及这如何影响训练。
- Pipeline 自定义指导：如何安全地交换机器人/URDF，并调整边界、步长和选项（如 IK 初始化）。

### 绝对 vs 相对 EE 控制

本指南中的示例使用绝对末端执行器（EE）姿态，因为它们易于理解。在实践中，相对 EE 增量或关节位置通常作为学习特征更受欢迎。

使用 processor，您可以选择要用于策略的学习特征。这可以是关节位置/速度、绝对 EE 或相对 EE 位置。您还可以选择存储其他特征，例如关节扭矩、电机电流等。

## 三个 Pipeline

我们经常组合三个 pipeline。根据您的设置，如果 action 和 observation 空间已经匹配，某些可以为空。
这些 pipeline 中的每一个都处理不同 action 和 observation 空间之间的不同转换。以下是每个 pipeline 的快速说明。

1. Pipeline 1：遥操作 action 空间 → 数据集 action 空间（手机姿态 → EE 目标）
2. Pipeline 2：数据集 action 空间 → 机器人命令空间（EE 目标 → 关节）
3. Pipeline 3：机器人 observation 空间 → 数据集 observation 空间（关节 → EE 姿态）

以下是我们在手机到 SO-100 follower 示例中使用的三个 pipeline 的示例：

```python
phone_to_robot_ee_pose_processor = RobotProcessorPipeline[RobotAction, RobotAction]( # 遥操作 -> 数据集 action
    steps=[
        MapPhoneActionToRobotAction(platform=teleop_config.phone_os),
        EEReferenceAndDelta(
            kinematics=kinematics_solver, end_effector_step_sizes={"x": 0.5, "y": 0.5, "z": 0.5}, motor_names=list(robot.bus.motors.keys()),
        ),
        EEBoundsAndSafety(
            end_effector_bounds={"min": [-1.0, -1.0, -1.0], "max": [1.0, 1.0, 1.0]}, max_ee_step_m=0.20,
        ),
        GripperVelocityToJoint(),
    ],
    to_transition=robot_action_to_transition,
    to_output=transition_to_robot_action,
)

robot_ee_to_joints_processor = RobotProcessorPipeline[RobotAction, RobotAction]( # 数据集 action -> 机器人
    steps=[
        InverseKinematicsEEToJoints(
            kinematics=kinematics_solver, motor_names=list(robot.bus.motors.keys()), initial_guess_current_joints=True,
        ),
    ],
    to_transition=robot_action_to_transition,
    to_output=transition_to_robot_action,
)

robot_joints_to_ee_pose = RobotProcessorPipeline[RobotObservation, RobotObservation]( # 机器人 obs -> 数据集 obs
    steps=[
        ForwardKinematicsJointsToEE(kinematics=kinematics_solver, motor_names=list(robot.bus.motors.keys()))
    ],
    to_transition=observation_to_transition,
    to_output=transition_to_observation,
)
```

## 为什么使用 to_transition / to_output

要从机器人/遥操作器转换到 pipeline 并返回，我们使用 `to_transition` 和 `to_output` pipeline 适配器。
它们标准化转换以减少样板代码，并在机器人和遥操作器的原始字典与 pipeline 的 `EnvTransition` 格式之间架起桥梁。
在手机到 SO-100 follower 示例中，我们使用以下适配器：

- `robot_action_to_transition`：将遥操作 action 字典转换为 pipeline transition。
- `transition_to_robot_action`：将 pipeline transition 转换为机器人 action 字典。
- `observation_to_transition`：将机器人 observation 字典转换为 pipeline transition。
- `transition_to_observation`：将 pipeline transition 转换为 observation 字典。

查看 [src/lerobot/processor/converters.py](https://github.com/huggingface/lerobot/blob/main/src/lerobot/processor/converters.py) 了解更多详细信息。

## 数据集特征契约

数据集特征由保存在数据集中的键确定。每个步骤都可以在名为 `transform_features(...)` 的契约中声明它修改的特征。一旦构建了处理器，处理器就可以使用 `aggregate_pipeline_dataset_features()` 聚合所有这些特征，并使用 `combine_feature_dicts(...)` 合并多个特征字典。

以下是我们在手机到 SO-100 follower 示例中如何使用 `transform_features` 方法声明特征的示例：

```python
def transform_features(
    self, features: dict[PipelineFeatureType, dict[str, PolicyFeature]]
) -> dict[PipelineFeatureType, dict[str, PolicyFeature]]:
    # 我们只在数据集中使用 ee 姿态，所以我们不需要关节位置
    for n in self.motor_names:
        features[PipelineFeatureType.ACTION].pop(f"{n}.pos", None)
    # 我们指定此步骤的数据集特征，这些特征将存储在数据集中
    for k in ["x", "y", "z", "wx", "wy", "wz", "gripper_pos"]:
        features[PipelineFeatureType.ACTION][f"ee.{k}"] = PolicyFeature(
            type=FeatureType.STATE, shape=(1,)
        )
    return features
```

在这里，我们声明在此步骤中修改的 PolicyFeature，因此当我们运行 processor 时，我们知道可以期望什么特征。然后可以聚合这些特征并用于创建数据集特征。

以下是我们在手机到 SO-100 记录示例中如何聚合和合并特征的示例：

```python
features=combine_feature_dicts(
    # 运行 pipeline 的特征契约
    # 这告诉您在 pipeline 步骤之后特征会是什么样子
    aggregate_pipeline_dataset_features(
        pipeline=phone_to_robot_ee_pose_processor,
        initial_features=create_initial_features(action=phone.action_features), # <- 我们可以期望的 Action 特征，这些来自我们的遥操作设备（手机）和 action processor
        use_videos=True,
    ),
    aggregate_pipeline_dataset_features(
        pipeline=robot_joints_to_ee_pose,
        initial_features=create_initial_features(observation=robot.observation_features), # <- 我们可以期望的 Observation 特征，这些来自我们的机器人和 observation processor
        use_videos=True,
        patterns=["observation.state.ee"], # <- 在这里，您可以选择性地过滤我们想要存储在数据集中的特征，使用特定模式
    ),
),
```

工作原理：

- `aggregate_pipeline_dataset_features(...)`：在 pipeline 中应用 `transform_features` 并按模式过滤（当 `use_videos=True` 时包括图像，当指定 `patterns` 时包括状态特征）。
- `combine_feature_dicts(...)`：合并多个特征字典。
- 使用 `record_loop(...)` 记录使用 `build_dataset_frame(...)` 构建与 `dataset.features` 一致的 frame，然后我们调用 `add_frame(...)` 将 frame 添加到数据集。

## 自定义机器人 Pipeline 的指导

您可以存储以下任何特征作为您的 action/observation 空间：

- 关节位置
- 绝对 EE 姿态
- 相对 EE 增量
- 其他特征：关节速度、扭矩等

选择您想要用于策略 action 和 observation 空间的内容，并相应地配置/修改 pipeline 和步骤。

### 不同的机器人

- 您可以轻松重用 pipeline，例如，要将另一个机器人与手机遥操作一起使用，修改示例并交换机器人 `RobotKinematics`（URDF）和 `motor_names` 以将您自己的机器人与手机遥操作一起使用。此外，您应该确保 `target_frame_name` 指向您的夹爪/手腕。

### 安全第一

- 更改 pipeline 时，从严格的边界开始，在使用真实机器人时实施安全步骤。
- 建议首先从仿真开始，然后转到真实机器人。
